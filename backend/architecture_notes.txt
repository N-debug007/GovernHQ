These are current thoughts on how we can structure the backend before building heavily.

High-Level Flow
---------------
User -> Frontend -> Backend API -> Database

The backend should be for agent state and logs so everything stays consistent.
The frontend should only talk to the backend through API endpoints.

Agent activity may come from two places:
- Orchestrator workflows (n8n, Zapier, etc.) via webhooks
- Fully autonomous agents running inside the backend

Regardless of the source, all activity should pass through the same ingestion flow so logging and future policy checks stay consistent.

If we add ML later for risk scoring or anomaly detection, it should plug into the backend without changing the overall structure.

Proposed Backend Structure
--------------------------
Very simple layout:

- api/      Routes and endpoint definitions
- services/ Business logic and ingestion flow
- models/   Database entities (ORM models)
- schemas/  Request/response validation and serialization
- core/     Shared configuration and utilities
- integrations/ Source-specific adapters

This keeps routing separate from logic and keeps database code out of the API layer.
It should also make it easier for multiple people to work without stepping on each other.
We can add more folders later only if the scope grows.

Scope (Iteration 1)
-------------------
For the first working iteration, the backend should support:

- Create agents
- List agents
- Log agent activity
- Disable an agent
- Retrieve logs for an agent

For orchestrator-driven agents, activity will enter through webhook endpoints.
For autonomous agents, activity can be handled internally but will still follow the same ingestion pipeline.

Anything beyond this (policy enforcement, analytics, dashboards, anomaly detection) can wait.
Priority should be a working end-to-end system before advanced features.

Database
--------
A relational database makes the most sense for now.
Agents and logs clearly have a relationship, and there is no strong need for NoSQL or vector storage at this stage.

API Draft (Initial Endpoints)
-----------------------------
- GET  /health
- GET  /agents
- POST /agents
- POST /agents/{id}/disable
- GET  /agents/{id}/logs

Webhook endpoints for orchestrator integrations can be added alongside these once ingestion is wired.

This gives the frontend a clear contract to build against.
