These are current thoughts on how we can structure the backend before building heavily.

High-Level Flow
---------------
User -> Frontend -> Backend API -> Database

The backend should be the source of the agent state and logs so everything stays consistent.
The frontend should only talk to the backend through API endpoints.
If we add ML later for risk scoring or anomaly detection, it should plug into the backend without changing the overall structure.

Proposed Backend Structure
--------------------------
Keep a simple layered layout:

- api/      Routes and endpoint definitions
- services/ Business logic and use cases
- models/   Database entities (ORM models)
- schemas/  Request/response validation and serialization
- core/     Shared configuration and utilities

This keeps routing separate from logic and keeps database code out of the API layer.
It should also make it easier for multiple people to work without stepping on each other.
We can add more folders later only if the scope grows.

Scope (Iteration 1)
-------------------
For the first working iteration, the backend should support:

- Create agents
- List agents
- Log agent activity
- Disable an agent
- Retrieve logs for an agent

Anything beyond this (policy enforcement, analytics, dashboards, anomaly detection) can wait.
Priority is a working end-to-end system before advanced features.

Database
--------
A relational database makes the most sense for now.
Agents and logs clearly have a relationship, and we do not need NoSQL or vector storage at this stage.

API Draft (Initial Endpoints)
-----------------------------
- GET  /health
- GET  /agents
- POST /agents
- POST /agents/{id}/disable
- GET  /agents/{id}/logs

This gives the frontend a clear contract to build against.
